# Logging Architecture

Swarpius has two parallel logging systems that share a common request ID for cross-referencing.

## Request IDs

Format: `rq-cNN-NNNN` (e.g. `rq-c01-0003`)

- `cNN`: conversation counter. When `ENABLE_DIAGNOSTIC_AGENT="true"`, the diagnostic agent classifies each request and reuses an existing `cNN` for continuations or mints a new one for topic changes. When disabled, it increments on idle gap exceeding `CONVERSATION_IDLE_TIMEOUT_SECONDS` (default 5 min) or on server restart.
- `NNNN`: monotonic request sequence within the conversation.

These are generated by `RequestIdGenerator` in `app/runtime/request_logger.py` (which delegates conversation assignment to `ConversationTracker`) and propagated via `contextvars` (`app/runtime/request_context.py`) so any code in the call stack can read it without parameter threading. On startup, counters are resumed from existing log directories so a restart does not overwrite logs.

## Conversation Logs

**Location:** `<SWARPIUS_DATA_DIR>/logs/conversation/<date>/<cNN>/<request-id>/` (default data dir is `agent/data/`).

High-level structured logs for each request are written by `RequestLogger`. With the diagnostic agent enabled, each conversation directory also gets a `conversation_summary.json` (topic + request list) and a `context_snapshot.json` (first-request-wins state capture).

```
logs/conversation/2026-03-26/c01/rq-c01-0001/
  request.json              — user input, timestamp, run mode
  outcome.json              — final status, chat response, timing, usage;
                              topic_summary + assignment_source when diagnostic agent active
  events.jsonl              — chronological WS emission stream
  coordinator_steps/
    step_01.yaml            — coordinator I/O + context snapshot per step
  tool_executions/
    01_roon_search.json     — full tool I/O, timing, retry details
  prompts/
    coordinator_system.txt
    context_providers_step_01.yaml
```

**Retention:** `LOG_RETENTION_DAYS` env var (default 7). Cleaned up on startup.

## Server Logs

**Location:** `<SWARPIUS_DATA_DIR>/logs/server/<date>/<cNN>/<request-id>/server.yaml`

Low-level YAML traces of the browse/action system. One file per request, grouped by conversation. Written by `ServerLogger` in `app/runtime/server_logger.py`.

Each entry is a YAML document separated by `---`:
```yaml
---
ts: '2026-03-26T14:30:00.123'
request_id: rq-c01-0003
op: resolve_ref_start
ref_id: '00007'
title: Aces High
```

### Operations logged

The main ops are listed below — not exhaustive. Other ops emitted include `resolve_ref` (when no reference is found), `position_session_failed`, `position_lookup_miss`, the `category_correction_*` family, `action_title_ref_mismatch`, `action_reference_recovered`, and `action_reset_to_root`. Grep for `get_server_logger().log(` to enumerate the full set.

| Op | Source | Details |
|---|---|---|
| `browse_core` | browse.py | aux, session_key, list_title, list_hint, item_count, items (first 10) |
| `drill_down` | browse.py | item_title, item_key, parent_path, result_count, depth_delta |
| `compile_ref` | browse.py | ref_id, reused (bool), title, hint, item_key, item_key_path |
| `resolve_ref_start` | browse.py | ref_id, title, cached_item_key, session_key, item_key_path |
| `resolve_ref_done` | browse.py | ref_id, tier (key_live/semantic_recovery/failed), success |
| `position_session_start` | browse.py | ref_id, session_key, parent_positions, target_position |
| `position_walk_step` | browse.py | index, position, fresh_key |
| `position_done` | browse.py | ref_id, cached_item_key |
| `get_actions_depth` | browse.py | ref_id, depth, list_hint, list_title, item_titles, item_hints, item_keys |
| `get_actions_drill` | browse.py | ref_id, depth, target_title, target_hint, target_item_key |
| `get_actions_done` | browse.py | ref_id, final_list_hint, final_list_title, action_titles, levels_pushed |
| `pick_drill_target` | browse.py | reason (no_items/duplicate/gateway_match/gateway_mismatch/uniform_group/no_match), chosen, plus reason-specific extras |
| `action_get_actions` | roon_action.py | item_title, item_ref, list_hint, list_title, action_titles, levels_pushed, recovered |
| `action_execute` | roon_action.py | item_title, action, action_item_key, session_key |

### Cross-referencing

Server logs and conversation logs for the same request share parent directories (`<date>/<cNN>/<request-id>/`), so they sit side-by-side:

```
data/logs/server/2026-03-26/c01/rq-c01-0003/server.yaml
data/logs/conversation/2026-03-26/c01/rq-c01-0003/
```

View the server trace for a request directly:

```bash
cat data/logs/server/2026-03-26/c01/rq-c01-0003/server.yaml
```

### Adding new log points

```python
from app.runtime.server_logger import get_server_logger
get_server_logger().log("my_operation", key1="value1", key2=42)
```

The request_id is injected automatically from contextvars. No need to pass it explicitly.

**Retention:** Same as conversation logs (`LOG_RETENTION_DAYS`). Cleaned up on startup.

## Test behaviour

Tests use `NullServerLogger` (injected via `conftest.py`) and `NullRequestLogger` to prevent any disk writes during test runs.
